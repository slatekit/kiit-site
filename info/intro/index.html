Slate Kit

Slate Kit is a Kotlin based open-source set of libraries. It is comprised of several projects and modular components that collectively provide pre-built architecture that can be used to build any type of application on the JVM. Slate Kit can also be used as an open-source alternative to FireBase/Parse Beta* as the server-side backend for your mobile / web applications.

When was it released?

1 2015 Started off in an idea in 2015. Slate Kit was being built in-parallel with a mobile app. The idea was to keep the architecture side as decoupled from the business/mobile app as possible.
2 2016 Actively builing out the utilities and architecture components for both client ( Android ) and Server. Started separating code out into various projects for better structure and organization
3 2018 Actively used in production and in beta apps. Dog-fooding for 1.2 years to help polish the components.
4 2019 Build automation in place, distribution to BinTray, Unit-Tests, Documentation updated

Why would you use it?

1 Features Several architecture components and features are pre-built and many of them are simple, pragmatic, effective and usable on both the client ( Android ) and Server.
2 Time It can take several months to years to build out an new system, architecture, and foundation. Leverage some existing best-practices, and reusable code to shorten this to a fraction of the time
3 Money Building software can be expensive. When building a new product/business, you should spend as much time as you can focusing on the business, not the technology.
4 Flexible Provides low-vendor lockin for leveraging some cloud services and operations like File Storage (AWS S3), Queues ( AWS SQS), using abstractions so you can easily switch between vendors.

How do you use it?

### Gradle 
    repositories {
        maven { url  "https://dl.bintray.com/codehelixinc/slatekit" }
    }

    dependencies {
        // other libraries
        
        // slatekit-results: Result<T,E> to model successes/failures with optional status codes
        compile 'com.slatekit:slatekit-results:1.0.0'
        
        // slatekit-common: Utilities for Android or Server
        compile 'com.slatekit:slatekit-common:1.0.0'
        
        // Misc architecture components ( all depend on results/common components above )
        // You can just use the project/module that you need
        compile 'com.slatekit:slatekit-app:0.9.24'
        compile 'com.slatekit:slatekit-cli:0.9.24'
        compile 'com.slatekit:slatekit-cloud:0.9.24'
        compile 'com.slatekit:slatekit-jobs:0.9.24'
    }
There are many components availables, but the most basic and general purpose for both client and server are the Results and Utilities components.
1 Results Models successes and failures using optional status codes. Similar to Result type in other langauges but with 1-2 major differences. Check out the Results page
2 Utilities General purposes utilities such as environment selection, configs, validation and much more. Check out the Utils page
3 Components There are several architecture components available. You can use as little or as much as you need. Check out all them modules available further below and on the Components page.
4 Examples Example are available for every utility and module. Click on the corresponsding utilities / components for details documentation. You can also see the source code for all examples here: Examples

Compare

Since Slate Kit is 100% Kotlin and runs on the JVM, it only makes sense to compare similar technologies.
1 Spring Spring is a comprehensive, battle tested framework for Java, with multiple modules available for building enterprise applications. The main differences between Spring and Slate Kit are:
  1. Simpler with fewer modules
  2. 100% Kotlin
  3. Functional/Immutable
  4. Many components can be used for both Android + Server
  5. Ideal for smaller projects/start-ups
  6. Leverages existing Kotlin http server ( Ktor )
2 Vert.x Vert.x is another comprehensive framework and is also available for many different JVM languages. The main differences between Vert.x and Slate Kit are similar as for Spring

Philosophy

Tech Built for the JVM, with Cloud Integration, and designed to be deployed and hosted anywhere.
Design Library based approach instead of a "Framework". Use as little or as much as you need.
Standards Check our coding standards
Approach Simplicity and Practicality above all else
Components Composable, single-purpose components as building blocks. Organized into various projects.
Portable Designed for low vendor lock-in via a "library" based approach. Easy to get out of this tool-kit
Minimal Dependencies on external, 3rd-party libraries are kept to a minimum
Open Source Open-Source under Apache 2.0 ( majority of components ) and/or LGPLv3.
License Apache 2.0 and LGPLv3: Please check our git repo for more info
Style Emphasis on immutability and functional programming as much as possible.
However, this is NOT a 100% purely functional code base.
We also carefully avoid certain language features in specific times/places in favour of simple, understandable, maintainable code.
